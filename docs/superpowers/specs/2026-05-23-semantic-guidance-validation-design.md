# 语义增强协同引导验证程序与标注工具设计

## 1. 背景与目标

本设计面向论文《基于语义增强的协同侦察引导技术》的实验复现需求，优先服务以下目标：

1. 复现论文中的核心验证逻辑，即在不同定位误差条件下，对比传统坐标引导方法与语义增强引导方法的锁定成功率差异。
2. 产出可重复、可追溯、可直接用于答辩图表制作的实验结果。
3. 提供一套轻量人工标注工具，将现有场景图片转换为结构化语义数据，作为实验程序输入。

本轮设计不追求复刻 CARLA、ROS、MQTT 等系统级联调环境，而是构建“模块化轻量仿真程序 + 浏览器单页标注器”的最小可用验证体系。

本项目定位为科研建议型原型，而非交付型工程。设计与实现优先级遵循以下原则：

1. 优先快速得到可运行、可解释、可复现实验结果。
2. 优先保留研究灵活性，而非追求工程完备性。
3. 优先简化开发流程，避免引入不必要的架构层次、自动化和重型依赖。

## 2. 范围界定

### 2.1 本轮纳入范围

1. 轻量仿真实验程序。
2. 纯规则语义增强判别器。
3. 基于图片的人工标注工具。
4. 最小可用的实验结果汇总、统计和图表输出。

### 2.2 本轮不纳入范围

1. CARLA/ROS/MQTT 实时联调。
2. 真实 VLM 或在线 GPT API 接入。
3. 自动检测、自动预标、多人协作标注。
4. 面向通用数据集的复杂标注平台功能。

## 3. 总体方案

整体方案采用“够用即可”的模块化轻量仿真架构，主链路如下：

1. 读取图片标注数据，构建实验场景库。
2. 基于真值目标位置注入定位误差，生成带偏差的粗引导坐标。
3. 在支援视角中生成候选目标集合。
4. 分别运行传统坐标引导方法和语义增强引导方法。
5. 根据真值目标判断锁定成功、误锁或未锁定。
6. 汇总多轮实验结果，输出统计表与图形结果。

程序采用共享场景、共享误差样本、共享评估标准的对照模式，确保不同方法之间的比较公平。

第一版实现上不强制追求完整分层，只要求逻辑边界清楚、后续便于替换评分规则和补充实验项即可。

## 4. 系统组成

### 4.1 场景库模块

职责：

1. 扫描 `data/images` 及其对应标注文件。
2. 为每张图片构造标准化场景对象。
3. 提供场景级属性，包括目标对象、干扰对象、地标、关系和场景标签。

输入：

1. 图片文件。
2. 同名 JSON 标注文件。

输出：

1. 内存中的 `SceneRecord` 列表。

### 4.2 误差注入模块

职责：

1. 对真值目标位置叠加二维高斯噪声。
2. 支持论文中的四档误差等级：`5m`、`15m`、`30m`、`50m`。
3. 基于随机种子保证实验可复现。

输入：

1. 真值目标位置。
2. 误差等级。
3. 随机种子。

输出：

1. 带偏差的粗引导位置。
2. 本轮误差向量记录。

### 4.3 候选集生成模块

职责：

1. 根据粗引导位置和预设视场规则，筛出候选目标集合。
2. 输出传统方法和语义方法共同使用的候选对象。

设计约束：

1. 第一版不做复杂成像模型。
2. 采用简化几何规则或基于标注框中心点的局部筛选。
3. 同一实验轮次中，两种方法共享同一候选集合。

### 4.4 传统坐标引导模块

职责：

1. 实现论文中的传统对照方法。
2. 根据粗引导坐标进行几何最近原则选目标。

第一版策略：

1. 以目标框中心到粗引导点的距离为主要评分。
2. 选择距离最小的候选对象作为锁定结果。

### 4.5 语义增强引导模块

职责：

1. 实现不依赖真实 VLM 的规则型语义增强方法。
2. 对每个候选对象计算综合语义得分。

第一版评分项：

1. `coordinate_score`：候选对象与粗引导位置的一致性。
2. `category_score`：对象类别是否符合预期。
3. `attribute_score`：对象属性标签是否符合目标描述。
4. `relation_score`：对象与地标的相对关系是否符合标注语义。
5. `scene_score`：场景标签是否与目标描述一致。

综合得分采用加权求和。第一版允许直接写在简洁配置文件中，或先以内置默认值实现，后续再外提配置。

### 4.6 评估与报告模块

职责：

1. 记录每次实验明细。
2. 统计各误差档下的方法表现。
3. 生成论文复现所需图表和表格。

核心输出：

1. 实验明细 CSV。
2. 方法级统计汇总 CSV。
3. 成功率对比折线图。
4. 误锁率对比图。
5. 可用于答辩汇报的摘要表。

## 5. 单次实验流程

单次实验以一个场景、一种误差等级和一个随机种子为单位，流程如下：

1. 读取当前场景的真值目标与对象列表。
2. 根据误差等级对真值目标位置注入二维高斯噪声。
3. 生成粗引导坐标。
4. 根据粗引导坐标筛选候选对象集合。
5. 运行传统坐标引导方法，得到方法 A 的锁定对象。
6. 运行语义增强引导方法，得到方法 B 的锁定对象。
7. 将两种方法输出分别与真值目标比对，记录成功锁定、误锁或未锁定。
8. 保存本轮实验明细，包括误差向量、候选集合、各评分项和最终判定。

## 6. 实验配置设计

实验配置建议采用独立配置文件，包含以下内容：

1. 参与实验的场景列表。
2. 误差等级列表。
3. 每档误差的重复实验次数。
4. 随机种子生成规则。
5. 各评分项权重。
6. 判定阈值。
7. 输出目录。

建议支持至少两种运行模式：

1. `full`：完整批量实验，生成论文主结果。
2. `debug`：少量场景快速试跑，用于调试规则与权重。

若实现期时间紧张，第一版可先只做 `debug` 模式和单一批处理入口，待实验跑通后再补 `full` 模式。

## 7. 数据格式设计

### 7.1 图片标注文件命名

每张图片对应一个同名 JSON 文件。

示例：

1. `a-01.png`
2. `a-01.json`

### 7.2 标注 JSON 结构

```json
{
  "image": "data/images/a-01.png",
  "scene_id": "a-01",
  "scene_tags": ["airport", "day", "open_area"],
  "target_object_id": "tank_1",
  "objects": [
    {
      "id": "tank_1",
      "type": "target",
      "label": "tank",
      "bbox": [460, 700, 280, 150],
      "attributes": ["left", "near_road"]
    },
    {
      "id": "tank_2",
      "type": "distractor",
      "label": "tank",
      "bbox": [1450, 640, 220, 120],
      "attributes": ["right", "near_hangar"]
    },
    {
      "id": "hangar_1",
      "type": "landmark",
      "label": "hangar",
      "bbox": [1670, 430, 330, 360],
      "attributes": ["right_side"]
    }
  ],
  "relations": [
    {
      "subject_id": "tank_1",
      "predicate": "left_of",
      "object_id": "hangar_1"
    },
    {
      "subject_id": "tank_2",
      "predicate": "near",
      "object_id": "hangar_1"
    }
  ],
  "notes": ""
}
```

### 7.3 第一版关系词表

第一版仅支持以下固定关系词：

1. `left_of`
2. `right_of`
3. `near`
4. `in_front_of`
5. `behind`

这样可以降低标注复杂度，并保证规则评分逻辑稳定。

## 8. 浏览器单页标注器设计

### 8.1 目标

标注器的目标是让用户以尽量低的学习成本，为每张场景图生成实验可用的结构化标注。

### 8.2 交互流程

第一版交互流程如下：

1. 自动读取 `data/images` 中的图片清单。
2. 显示当前图片并支持上一张、下一张、跳转。
3. 用户在图上框选对象。
4. 为每个框设置 `target`、`distractor` 或 `landmark`。
5. 填写对象 `id`、`label` 和可选属性。
6. 从对象列表中指定唯一真目标 `target_object_id`。
7. 填写 `scene_tags`。
8. 通过固定关系词创建对象关系。
9. 保存为同名 JSON。
10. 再次打开时自动回读已有标注。

### 8.3 第一版必须支持的功能

1. 图片浏览。
2. 画框、新增、删除、拖拽和缩放。
3. 对象类型与基本属性填写。
4. 真目标指定。
5. 场景标签填写。
6. 关系标注。
7. JSON 保存与重新加载。
8. 已标注状态与进度提示。

### 8.4 第一版暂不支持的功能

1. 自动预标。
2. 多人协作。
3. 通用数据集导出格式。
4. 复杂撤销历史。
5. 自动关系推断。

## 9. 错误处理与校验

程序需要提供基础校验，避免产生不可用数据，但不追求交付型产品的完整防呆能力。

### 9.1 标注器校验

1. 未指定 `target_object_id` 时禁止保存。
2. 存在空 `id` 或重复 `id` 时提示修正。
3. 关系引用不存在的对象时禁止保存。
4. 框选数量为零时允许保存草稿，但标记为未完成。

### 9.2 实验程序校验

1. 标注文件缺失时跳过对应图片并记录告警。
2. 真目标不存在时终止该场景实验。
3. 候选集合为空时记录为未锁定而非崩溃。
4. 未识别的关系词或标签按配置报错。

## 10. 测试与验证策略

### 10.1 标注器验证

1. 能正确读取图片列表。
2. 能创建和修改框。
3. 能保存 JSON 并重新加载。
4. 能拦截关键缺失项。

### 10.2 实验程序验证

1. 固定随机种子时结果稳定复现。
2. 同场景下传统方法与语义方法共享候选集。
3. 误差等级切换时结果符合预期趋势。
4. 报告输出能支撑人工复核即可，不要求复杂报表体系。

### 10.3 结果合理性验证

目标不是机械复现论文中的数值，而是验证相同趋势：

1. 小误差条件下两种方法差距较小。
2. 误差增大时传统方法性能更快下降。
3. 语义增强方法在高误差条件下保持更高成功率。

## 11. 交付物

本设计最终对应以下原型产物：

1. 轻量仿真实验程序源码。
2. 浏览器单页标注器源码。
3. 少量示例标注数据。
4. 最小配置文件或默认参数入口。
5. 实验结果 CSV。
6. 至少一张成功率对比图。
7. 一份用于答辩复述的简化实验说明。

## 12. 后续扩展路线

当前设计刻意保留了扩展空间，后续可逐步增加：

1. 真实 GPT 或在线 VLM 接口适配层。
2. 评分项权重搜索与消融实验。
3. 时延补偿与运动预测模拟。
4. 面向答辩的可视化实验回放界面。

当前不把这些扩展放入第一版，是为了先确保论文验证主链路尽快可落地、可运行、可展示。

## 13. 简化开发策略

为贴合科研建议型项目定位，第一版实现建议刻意简化：

1. 先做“单人本地可用”而不是“可部署可分发”。
2. 先做“少量样本跑通”而不是“一次性全量完成”。
3. 先做“代码直白易改”而不是“架构高度抽象”。
4. 先做“图上手工标注 + 规则评分”而不是“模型接入 + 自动推理”。
5. 先做“结果趋势成立”而不是“数值完全逼近论文表格”。

对应到开发顺序，建议采用如下最短路径：

1. 先实现标注器最小闭环。
2. 先标 3 到 5 张代表性图片验证标注格式。
3. 再实现单场景、单误差档实验脚本。
4. 跑通后再扩展到多场景、多误差档批量统计。
5. 最后再补答辩图表和说明材料。
